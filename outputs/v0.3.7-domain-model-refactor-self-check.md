# v0.3.7 Domain Model Refactor Self Check

Date: 2026-07-12

## Scope

- Split the oversized `App.tsx` without changing the visible product workflow.
- Establish inventory, recipe, meal-plan, and diary domain models.
- Create clear boundaries for the v0.4.0 persistence work.

## Result

- Reduced `App.tsx` from 2,244 lines to 308 lines.
- Added four domain modules under `src/domain/`.
- Added separate feature modules for Home, Upload, Fusion, Recipes, and Diary.
- Moved static food, tool, preference, recognition, and recipe seed data to `src/data/catalog.ts`.
- Moved reusable visual primitives to `src/components/ui.tsx`.
- Removed inactive inventory-browser components left over from the pre-fusion Warehouse design.

## Dependency Direction

1. `domain` contains data contracts and pure business rules.
2. `data` provides the current in-memory catalog and seed records.
3. `features` renders one product capability and consumes domain/data modules.
4. `App.tsx` owns global state transitions and composes feature screens.

Domain modules do not import React or feature components.

Manual smoke path:

1. Home -> Upload Ingredient -> complete upload -> Warehouse.
2. Warehouse -> select ingredient -> fuse -> favorite -> Today.

## Product Judgment

This refactor is complete only as an architecture baseline. It does not make Miiix a usable MVP by itself. The next version must add persistence for inventory, saved recipes, today's plan, shopping list, and diary records before new visual features are accepted.
