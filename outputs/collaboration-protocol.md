# 冰箱融合器协作协议

日期：2026-06-25

## 1. 协作原则

用户希望获得严格辅导，但不希望被固定作业和连续提问拖住节奏。

因此后续协作采用：

- 我负责主动推进、做假设、给判断、产出文档/原型/代码。
- 用户负责在不同意时纠偏，而不是被动回答一串问题。
- 单个用户案例只作为产品线索，不作为产品定位变化的充分证据。
- 严格监管体现在范围控制、假设审查、验证设计和迭代节奏，而不是布置僵硬作业。

## 2. 默认工作方式

每次推进时，我默认输出：

- 当前判断
- 关键假设
- 产品/设计/技术动作
- 风险和反例
- 下一步我将直接做什么

不默认输出：

- 固定作业
- 连续提问
- 需要用户补充才能继续的阻塞式流程

## 3. 提问规则

默认不向用户提问。

只有以下情况才需要用户确认：

- 涉及账号、密钥、付费、发布、外部发送。
- 涉及删除、覆盖、迁移等不可逆操作。
- 产品方向发生重大改变。
- 存在多个高风险路线，且没有足够证据能做默认选择。

## 4. 防止单个案例放大

每次从用户输入中提取洞察时，必须标注其证据等级：

- L1：单个样本线索，只能作为灵感或假设。
- L2：多个样本反复出现，可以进入需求池。
- L3：通过访谈、数据或可用性测试验证，可以进入 MVP。

规则：

- L1 不能改变产品定位。
- L1 可以影响默认文案、测试样例、可选参数。
- MVP 主链路必须由核心问题驱动，而不是由单次表达驱动。

## 5. 当前产品主线

保持为：

> 冰箱库存管理 + 基于库存生成菜谱 + 做完后更新库存/购物清单。

创新菜式、地域口味、学习新菜系、一个人吃饭不将就，全部作为可选生成参数或后续增强能力，不作为当前产品本质。
