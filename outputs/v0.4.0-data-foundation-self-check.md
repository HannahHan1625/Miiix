# v0.4.0 Data Foundation Self Check

Date: 2026-07-13

## Scope

- Establish the target database Schema.
- Make database changes versioned and reviewable through migrations.
- Establish Repository contracts between product features and persistence infrastructure.
- Preserve the current user-facing prototype while preparing the persistence migration.

## Result

- Added four ordered PostgreSQL migrations under `database/migrations/`.
- Added canonical catalog, private user operations, recommendation evidence, and row-level security layers.
- Added normalized persistence entities in `src/domain/persistence.ts`.
- Added Repository contracts in `src/data/repositories/`.
- Added an atomic, idempotent inventory transaction function so balance updates and ledger writes cannot drift apart.
- Added embedding-dimension validation and hardened parent-child ownership checks in row-level security policies.
- Added `pnpm run check:migrations` to reject missing, malformed, or out-of-order migrations.
- Updated the product version to `v0.4.0 数据地基`.

## Dependency Direction

1. Product features depend on Repository interfaces.
2. Repository interfaces depend on domain persistence records.
3. A future Supabase adapter will implement the interfaces.
4. SQL migrations remain independent from React.

The React application does not import Supabase directly.

## Explicit Non-Claims

- No live Supabase project has been created or connected.
- No production credentials or AI provider secrets are stored in the repository.
- Current React state has not yet been migrated to Repository calls.
- The 200-ingredient reviewed seed catalog has not yet been built.
- Epicure embeddings have not yet been imported.
