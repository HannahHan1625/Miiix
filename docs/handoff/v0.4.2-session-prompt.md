### 三、继承的产品定义

Miiix 的产品本质已经确定：

> 面向有做饭习惯的独居者和小家庭，以食材库存管理和基于真实库存的菜谱生成，减少食材浪费与“今天吃什么”的决策成本。

核心闭环：

`录入食材 -> 查看库存 -> 选择食材/厨具/偏好 -> 生成候选菜谱 -> 今天做 -> 生成购物缺口 -> 开始制作 -> 完成制作 -> 扣减库存 -> 形成日记`

四项主导航已经确定：

- 主页：用户概览、统一录入入口、库存关注项。
- 仓库：选择库存食材、厨具和偏好并融合。
- 菜谱：全部、收藏、做过、菜谱反推。
- 日记：今日计划、制作、完成确认和日历账本。

菜谱生命周期已经确定：

- Candidate：候选。
- Favorite：收藏。
- Planned：今天做。
- Cooking：开始制作。
- Completed：完成并扣库存、写日记。
- Cancelled：取消或被替换。

严禁重新把“收藏”“今天做”“看做法”和“完成制作”混成同一状态。

### 四、v0.4.1 已完成的工程事实

v0.4.1 已跑通浏览器本地持久化纵向链路：

`新增库存 -> 采购流水 -> 今日计划 -> 购物清单 -> 制作会话 -> 库存消耗 -> 完成计划 -> 日记 -> 刷新恢复`

当前技术栈：

- React 19。
- TypeScript 5.7。
- Vite 6。
- IndexedDB + `idb`。
- Vitest + fake-indexeddb。
- 目标云数据库为 Supabase PostgreSQL，但尚未部署。
- Capacitor iOS 壳保留，但当前路线暂停 iOS。

当前分层：

- `src/features/`：页面和交互。
- `src/application/`：业务用例编排。
- `src/domain/`：纯领域模型和规则。
- `src/data/repositories/contracts.ts`：Repository 合同。
- `src/data/repositories/indexeddb/`：IndexedDB Adapter。
- `src/App.tsx`：Composition Root，在这里选择具体 Provider。
- `database/migrations/`：目标 Supabase PostgreSQL Schema。

必须保持依赖方向：

`Feature/UI -> Application -> Domain + Repository contracts <- Adapter`

Application 和 Domain 不得导入具体 IndexedDB Store、Supabase Client 或 SQL。具体 Adapter 只在组装入口选择。

v0.4.1 事务和幂等规则必须保持：

- 完成制作、库存扣减、计划完成和日记投影必须原子提交。
- 制作幂等键为 `meal-plan:{planId}`。
- 消耗幂等键为 `cook:{sessionId}:{lotId}`。
- 重复完成同一计划不得二次扣库存。
- UTC 时间戳和用户本地业务日期必须分开处理。

### 五、v0.4.2 的唯一主目标

版本名称：`v0.4.2 食材主数据`。

唯一主目标：

> 建立经过审核、可机器校验、可导入和可查询的首批中国家庭高频食材标准目录，使库存、识别、菜谱、图片和未来模型共享稳定食材 ID。

### 七、建议实施阶段

严格按以下顺序推进，不要一次性生成 200 行看似完整但无法验证的数据。

#### 阶段 1：审查现状

- 盘点 `src/data/catalog.ts`、数据库 Catalog Schema、Repository 合同和 UI 当前依赖字段。
- 建立“现有字段 -> 目标字段 -> 使用页面 -> 数据来源”的矩阵。
- 找出字符串名称映射、重复定义、硬编码调味品和图片错配。
- 明确哪些字段属于公共主数据，哪些属于用户库存批次。

#### 阶段 2：定义数据标准

- 定义 ID 命名或 UUID 策略。
- 定义分类树和分类代码。
- 定义别名类型：口语、地区、零售、OCR、英文、外部模型。
- 定义储存方式、保质期上下文和安全提示边界。
- 定义数量模式、标准单位、默认购买份量和换算。
- 定义营养数据来源和版本。
- 定义图片授权和审核字段。
- 形成 JSON Schema 和字段说明。

#### 阶段 3：30 个黄金样本

- 选取真实高频且能覆盖字段差异的食材。
- 至少包含鸡蛋、鸡翅、鸡胸肉、猪肉、牛肉、茄子、青椒、番茄、土豆、桃、杨梅、豆腐、牛奶、虾、鱼、大米、面条、食用油、盐、生抽等典型对象。
- 不把示例清单机械当成最终 30 个；根据类别覆盖和数据可得性形成正式清单。
- 每项引用可信来源，不自由编造保质期、热量或平均价格。
- 价格若缺乏可持续来源，可将字段标记为可选或暂缓，不能伪造全国平均值。

#### 阶段 4：导入和查询

- 将黄金样本导入本地 Catalog Adapter。
- 保持 Repository 合同稳定；如果必须扩展，先解释业务原因和兼容影响。
- UI 只读取 Catalog 投影，不继续复制食材对象。
- 为别名搜索、分类过滤和储存默认值建立测试。

#### 阶段 5：质量门槛

- 运行 schema validation。
- 运行重复、冲突和外键检查。
- 抽检图片名称匹配和授权。
- 检查 390px 页面中数据投影没有破坏现有布局。
- 运行原有纵向测试，确保数据重构没有破坏 v0.4.1 闭环。
- 只有 30 个黄金样本通过后，才提出扩展到 200 个的计划。

### 八、目标食材字段基线

请在审查后确定最终 schema，但至少评估以下字段：

- `id`
- `canonicalNameZh`
- `canonicalNameEn`
- `scientificName` 或权威外部标识
- `categoryLevel1Id`
- `categoryLevel2Id`
- `categoryLevel3Id`
- `aliases[]`，每项包含名称、类型、语言/地区和来源
- `defaultStorageProfileId`
- `storageProfiles[]`，包含区域、环境标签、建议天数和来源
- `defaultUnitId`
- `supportedUnitIds[]`
- `defaultAmountMode`
- `defaultPurchaseQuantity`
- `unitConversions[]`
- `caloriesPer100g`
- `nutritionSource`
- `referencePrice`，如保留必须包含地区、日期、单位和来源
- `imageAssets[]`，包含 URL/路径、来源、许可、处理状态、主图和审核状态
- `externalMappings[]`，例如 Epicure token
- `status`
- `dataVersion`
- `createdAt`
- `updatedAt`
- `reviewedBy`

不要让公共目录直接包含用户实付价格、采购日期或库存余额；这些属于 Inventory Lot。

### 九、数据来源与版权规则

这是强制规则：

1. 优先权威、稳定、可引用的公开资料或明确许可数据。
2. 营养、储存和食品安全信息必须记录来源和抓取/审核日期。
3. 不能把电商平台页面内容或小红书图片批量抓取后当成可商用资产库。
4. 参考小红书只代表审美方向，不代表图片授权来源。
5. 图片必须记录许可状态；未知许可只能用于本地研究，不得进入发布资产。
6. AI 生成或编辑图片也要保存生成来源、Prompt/模型和人工审核状态。
7. 不要为了填满字段编造精确数据。缺失值可以显式为空并进入待补清单。
8. 外部模型映射必须可追溯，不能仅靠字符串相似度静默绑定。

如需要在线研究最新数据库、论文、模型或许可证，必须浏览并引用原始来源或官方文档。

### 十、工程边界

继续使用当前 React + TypeScript + Vite。不要迁移 Vue，不要引入 ECS，不要用 Three.js 重做仓库，不要参考 PocketJS 发起 runtime 重构。

Repository/Adapter 原则：
