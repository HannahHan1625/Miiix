## 下一数据会话的唯一目标

先定义“菜谱共现/搭配关系”的数据合同和证据门禁，不直接接模型、不抓取菜谱、不扩 UI：

- 菜谱来源、复制/再发布权利和 attribution；
- recipe identity 与重复菜谱合并规则；
- ingredient occurrence 的 Miiix ID、形态、用量、角色和步骤关系；
- 共现边方向、支持度、置信度、来源批次和版本；
- “三样食材”合理性、惊喜度、覆盖率、额外采购量的离线评测；
- 关系缺失或 pending 时的显式回退。

完成调研与合同后先向 Hannah 报告候选范围和风险，再决定它使用哪个版本号。不要自动把该工作塞进已规划的 `v0.4.3 可部署 Web`。

## 严禁误判

- Epicure token/embedding 能表达邻近关系，不等于 Miiix 已拥有可再发布菜谱库或已验证推荐质量。
- 模型和 vocabulary 的 CC BY 4.0 不自动覆盖底层 11 个异构菜谱来源。
- `published import batch` 可以包含已审核的 pending mapping，不等于全部 mapping approved。
- bought-as-ground 是 purchased lot；user-transformed lot 仍需独立原子转换用例。
- 生成菜谱图片后续采用 AI 生图；统一风格在 UX 试验后确定，本数据会话不生成生产图片。
- 不把 API Key 放进前端，不宣称 Supabase 已上线。
