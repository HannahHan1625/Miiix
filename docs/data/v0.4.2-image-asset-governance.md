# Miiix v0.4.2 食材图片资产治理

> 适用范围：v0.4.2 的 30 项食材主数据图片。
> 当前结论：30 项全部使用 **Miiix 自有的程序化 SVG 原型占位图**，不是照片，也不是 AI 生成图。
> 本文定义资产事实和发布门禁，不代表相关自动化测试已经完成。

## 1. 当前资产决策

v0.4.1 的食材卡片混用了外部图片 URL 和少量本地 SVG，但外部图片没有完整保存作者、原始页面、许可版本、主体核对和处理记录。v0.4.2 不对这些缺失信息进行猜测，也不把“图片可以打开”当作“可以发布”。

因此，当前 30 项统一使用由 Miiix 代码按食材名称和固定色板生成的 SVG：

- 不引用任何第三方照片；
- 不热链外部图片；
- `originalUrl/sourceUrl/providerAssetId` 均为 `null`；
- `licenseCode=MIIIX-OWNED`，`rightsStatus=cleared`；
- `processingStatus=placeholder`；
- `licenseStatus=approved_for_prototype`，不是生产照片的 `approved`；
- `styleConsistency=prototype_placeholder`；
- `aiGeneration=null`，明确不是 AI 生成；
- alt 文本写明“原型占位图，非实物照片”。

这使原型可以安全演示“浏览目录、录入库存、选食材、做计划”的交互闭环：画面稳定、无第三方版权缺口、30 项风格一致、食材名称可读。

但它不等于生产实拍素材：

- 占位图不能帮助用户判断鸡翅/鸡爪、猪肉/猪肉末等真实外观；
- 不能作为识别结果的视觉证据或模型训练数据；
- 不应在市场宣传中暗示为真实食材照片；
- 上线生产照片前仍需逐张完成来源、权利、主体和处理审核。

## 3. 资产 manifest 字段

manifest 不是一张只有 URL 的图片表。每个资产必须包含以下字段：

| 字段 | 必填规则 | 作用 |
|---|---|---|
| `id` | 固定 UUID，永不复用 | 资产身份 |
| `ingredientId` | 必须指向存在的食材 UUID | 防止鸡爪误挂鸡翅图 |
| `assetUri` | 应用可解析的受控 URI | 当前使用 `miiix-placeholder://...` |
| `assetType` | `source_photo / cutout / thumbnail` | 区分原图、抠图和缩略图 |
| `altText` | 非空，准确描述主体 | 无障碍与误用提示；占位图必须写“非实物照片” |
| `originalUrl` | 第三方或自摄原图时必填 | 回到原始证据页/文件 |
| `processedUrl` | 生成派生图时必填 | 定位实际发布文件 |
| `providerAssetId` | 外部图库素材时必填 | 防止仅靠会变化的 URL 定位 |
| `sourceUrl` | 第三方素材必填 | 保存素材详情页，不是搜索结果页 |
| `sourceSha256` / `processedSha256` | 有文件时必填 | 证明审核的文件和发布文件没有被替换 |
| `processingStatus` | 明确 source、processed 或 placeholder | 防止把占位图当成照片 |
| `transformLog[]` | 派生图和占位图必填 | 记录裁剪、抠图、压缩、色彩和生成方式 |
| `license` / `licenseCode` / `licenseUrl` | 第三方素材全部必填 | 保存具体许可，不写含混的“网络图片” |
| `rightsStatus` | cleared、pending 或 rejected | 是否有依据进入发布候选 |
| `licenseStatus` | approved、approved_for_prototype、pending 或 rejected | 区分生产发布与原型演示 |
| `attribution` | 许可要求署名时必填 | 提供实际展示的署名文案 |
| `subjectMatchReviewed` | 生产图片必须为 true | 人工确认品种、部位、生熟和形态 |
| `rightsReviewed` | 生产图片必须为 true | 人工确认许可与使用场景 |
| `styleConsistency` | 标记 prototype、consistent 或 needs_review | 控制 30 项视觉一致性 |
| `aiGeneration` | 非 AI 为 `null`；AI 图至少记录 model 与 prompt | 不能把 AI 图伪装成实拍或自有摄影 |
| `reviewStatus` | 只有 approved 可进相应发布范围 | 审核结论 |
| `isPrimary` | 每个 active 食材恰好一个 true | 卡片主图唯一 |

审核人、审核时间和目录 `dataVersion` 还应在资产所属记录或发布日志中保存。若以后加入 AI 图片，`model` 字符串应包含提供方和模型版本，生成时间、当时适用的服务条款和参考图权利写入 `transformLog`；后续版本可再拆成独立字段。

## 4. 两级发布门禁

### 4.1 原型占位图门禁

程序化 SVG 进入 v0.4.2 原型必须全部满足：

- 30 个 active 食材各有且只有一个 primary；
- `assetUri` 的 UUID 与 `ingredientId` 完全一致；
- `processingStatus=placeholder`；
- `licenseCode=MIIIX-OWNED`、`rightsStatus=cleared`；
- `licenseStatus=approved_for_prototype`，不能写生产 `approved`；
- `aiGeneration=null`；
- `originalUrl/sourceUrl/providerAssetId` 为 `null`，没有伪造第三方来源；
- alt 文本包含正式食材名和“原型占位图，非实物照片”；
- SVG 内容不执行脚本、不加载外部资源，且可在目标页面正常缩放；
- 每项 `subjectMatchReviewed=true`、`rightsReviewed=true`，这里表示名称绑定和自有代码权利已复核，不表示存在实拍主体。

### 4.2 生产照片门禁

真实照片、抠图或 AI 生成图要替换占位图，必须逐张满足：

1. **身份正确**：图片主体与正式食材粒度一致。鸡胸不能用鸡腿图，猪肉末不能用整块猪肉图，生大米不能用熟米饭图；
2. **来源可回溯**：保存原始详情页、提供方素材 ID、作者、取得日期、原始文件和 SHA-256；
3. **许可明确**：保存具体许可代码和条款 URL；许可允许产品的商业/非商业场景、修改、分发和应用内展示；
4. **署名可执行**：如果许可要求署名或相同方式共享，产品真实能履行，不能只在内部表格里记录；
5. **派生链完整**：原图、裁剪、抠图、描边、压缩和成品各自可定位，转换记录和成品 hash 完整；
6. **无额外权利风险**：检查人物肖像、商标、包装、菜品摆盘、受保护艺术品等非版权限制；
7. **审核完成**：`subjectMatchReviewed=true`、`rightsReviewed=true`、`reviewStatus=approved`，且生产 `licenseStatus=approved`；
8. **体验达标**：背景、视角、光线、裁切、清晰度和尺寸一致，弱网有本地降级，不依赖第三方热链；
9. **主图切换可回滚**：新图成为 primary 前保留旧资产记录，出现投诉时能立即切回自有占位图。

任何一项不满足，资产只能是 `pending/research_only`，不能靠“图库看起来免费”越过门禁。

## 5. 外部图库许可边界

### 5.1 Wikimedia Commons

[Wikimedia Commons 官方复用说明](https://commons.wikimedia.org/wiki/Commons%3AReusing_content_outside_Wikimedia/en) 明确指出：不同文件可能采用不同许可，可能要求作者署名、许可链接或相同方式共享；Commons 也不保证每个上传者的版权声明绝对正确，复用者仍需核验。

使用规则：

- 逐文件保存详情页、原作者、许可版本和 attribution，不把“来自 Commons”当成统一许可；
- CC BY-SA/GFDL 等许可的衍生要求必须和 Miiix 的发布方式兼容；
- 公有领域也保存来源和作者，便于追溯；
- 不直接热链；下载原文件、计算 hash，再进入处理链；
- 详情页有来源不明、争议、人格权或商标警告时不采用。

### 5.2 Unsplash

[Unsplash License](https://unsplash.com/license) 允许免费用于商业和非商业用途，通常不强制署名，但禁止未经实质修改出售图片，也禁止汇编图片来复制或竞争其服务。

使用规则：

- 即使许可不强制署名，也保存摄影师、照片详情页和素材 ID，产品可酌情展示感谢；
- 不把整批 Unsplash 图片包装成独立食材图库再分发；
- 不只保存 `images.unsplash.com` 参数 URL；必须保存对应详情页和取得时许可证据；
- 许可解决使用权，不解决“图里到底是鸡翅还是鸡腿”的主体审核。

v0.4.1 遗留 URL 没有这些 manifest 信息，因此不能自动标记 approved。

### 5.3 Pexels

[Pexels License](https://www.pexels.com/license/) 允许免费使用和修改，通常不要求署名；同时禁止出售未修改副本、暗示人物或品牌背书、在其他图库重新分发，以及把图片用于商标等。

使用规则：

- 保存摄影师、详情页、素材 ID 和取得时条款；
- 检查品牌包装、可识别人物和背书风险；
- 不把下载的原图原样作为可售素材或图库产品；
- 仍须通过食材主体、处理、hash 和主图门禁。

### 5.4 AI 生成或 AI 编辑图片

AI 图不是“没有作者所以自动无版权风险”。每张图必须：

- 在 `aiGeneration` 记录模型提供方/版本和完整 prompt；
- 记录生成时间、当时服务条款、商业使用范围和是否使用参考图；
- 使用参考图时，同时证明参考图允许生成衍生作品；
- 在 `transformLog` 记录扩图、去背景、重绘、修复和人工编辑；
- 人工检查是否生成错误食材部位、品牌包装、文字水印、人物或近似特定作品；
- 不把 AI 图标成 `source_photo` 或“实拍”；产品需要时明确披露为示意图；
- 仅凭模型提供方允许商业使用，不足以把 `rightsStatus` 自动设为 cleared，仍需人工权利审核。

当前 30 项 `aiGeneration=null`。程序生成的简单 SVG UI 占位图与生成式 AI 图片不是同一类资产。

### 5.5 明确禁止的来源

- 搜索引擎结果页、社交媒体转载、博客截图、商超商品页截图；
- 只有 CDN 图片 URL、找不到原始详情页的文件；
- 带水印、来源冲突或许可写“仅供个人/非商业使用”的素材；
- 无法证明参考图权利的 AI 编辑结果；
- 通过热链让第三方随时替换内容的 URL。

## 6. 抽检与复核规则

版权事实不能抽样：**每张第三方或 AI 资产都要逐张审核许可和来源**。抽检只用于补充发现视觉和流程问题，不能代替权利门禁。

### 6.1 v0.4.2 的 30 项

- 自动检查覆盖 100%：字段完整、UUID 引用、URI 规则、一个 primary、状态组合、占位图无外部 URL、AI 字段为空；
- 人工检查覆盖 100%：食材名称、alt 文本、颜色可读性、移动端裁切和“非实物照片”标识；
- 任意一项失败都阻止整批标记为原型可发布，不能用 29/30 通过率豁免。

### 6.2 扩展到 200 项或更大批次

- 来源与许可仍为 100% 逐张审核；
- 高风险项 100% 视觉复核：肉类部位、鱼虾品种、生熟对照、品牌包装、人物、AI 图、衍生编辑图；
- 其余视觉质量采用分层随机抽检：至少抽 20%，且不少于 30 张；每个一级分类、提供方、许可类型和处理流水线至少包含 3 张；
- 抽检发现 1 个关键错误（错食材、许可不成立、错署名、实拍/AI 标注错误），整批暂停并升级为 100% 复核；
- 一般质量错误率超过 2% 时，退回整个处理批次，修复后重新抽样；
- 主图变更后的 7 天内复核错误日志和用户反馈；提供方条款变化或收到权利投诉时立即重新审核受影响资产。

## 7. 生产实拍替换顺序

占位图不应推动团队一次性采购 200 张照片。建议按错误成本排序：

1. 先替换最容易误认且位于核心路径的 8 项：鸡翅、鸡胸、鸡爪、鸡腿、猪肉、猪肉末、大米（生）、米饭（熟）；
2. 再替换视觉明确的蔬果，用同一拍摄/处理规范校准风格；
3. 最后处理上位食材“鱼、虾、食用油、豆腐、牛奶”，先决定是展示通用示意图还是要求用户选到更细粒度；
4. 每一小批通过生产门禁后再切 primary，保留 SVG 占位图作为失败降级。

图片资产升级不能改变食材 UUID，也不能修改 Inventory Lot。图片只是食材的受治理展示资源，不是食材身份本身。
