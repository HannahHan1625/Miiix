# Miiix v0.4.2 现有字段到目标字段矩阵

> 对比基线：v0.4.1
> 目标：v0.4.2 食材主数据
> 用途：让产品、设计和工程共同知道“旧字段去哪里、哪个页面使用、依据是什么、如何迁移”。
> 本文是迁移口径，不代表迁移测试已经完成。

## 1. 怎么读这张表

- **现有**：v0.4.1 页面常量、`FoodInfo`、`InventoryItem` 或已有持久化对象中的字段；
- **目标**：v0.4.2 主数据、Repository 读模型或 Inventory Lot 中的字段；
- **使用页面**：该字段最终服务的用户界面；
- **来源**：谁对这个值负责；
- **处理**：升级时保留、迁移、重新审核、留空或删除的方式。

最重要的边界是：食材目录保存公共、可复核的默认事实；Inventory Lot 保存用户某一次真实购买和储藏事实。

## 2. 字段级矩阵

| 现有 v0.4.1 | 目标 v0.4.2 | 使用页面 | 来源 | 处理 |
|---|---|---|---|---|
| `FoodInfo.id`，如 `egg`、`pork`、`rice` | `IngredientDetail.id` 固定 UUID + `legacyIds[]` | 仓库、录入、菜谱、计划、购物清单 | Miiix 数据治理 | 旧 ID 建立一对一映射后迁移；UUID 永不复用。旧 `pork` 按原语义映射到“猪肉末”，旧 `rice` 映射到“米饭（熟）” |
| `FoodInfo.name` | `canonicalNameZh`；另有 `canonicalNameEn`、可空 `scientificName` | 所有食材展示与搜索 | Miiix 人工审核；外文名可参考来源记录 | 旧展示名不直接当真值；“肉沫”规范为“猪肉末”，原词转为已审核别名 |
| 没有 `legacyIds` | `legacyIds[]` | 用户无感；用于打开旧库存、旧菜谱 | 版本迁移清单 | 只做数据兼容，不参与普通别名搜索；一个旧 ID 只能指向一个 UUID |
| 没有 `kind`，页面靠分类猜食材角色 | `kind = raw / processed / condiment / beverage / dish_component` | 仓库筛选、菜谱主料/调料展示 | Miiix 领域规则 | 逐项人工赋值；不从中文字符串猜测 |
| `level1`、`level2`、`level3` 三个文本 | `categoryLevel1Id`、`categoryLevel2Id`、`categoryLevel3Id`、`categoryIds` | 仓库三级筛选、手动录入 | Miiix 固定分类树 | 文本迁移为 ID；校验三级父子链，不允许只有名字相同却挂错父级 |
| 页面内 `categoryTree` 数组 | `CatalogCategory` 独立记录：`id/parentId/slug/name/level/sortOrder/dataVersion` | 分类筛选、目录浏览 | Miiix 数据治理 | 由主数据生成 UI 投影；图标仍属展示层，不进入食材事实 |
| 没有别名表；部分页面直接比较 `name` | `aliases[]`：原词、标准化词、类型、语言/地区、来源、置信度、审核状态 | OCR/照片确认、手动搜索、菜谱导入 | Miiix 审核；外部模型标签需记录模型来源 | 只有 approved 别名可自动解析；`locale + normalizedAlias` 必须唯一；含混词返回候选而非静默命中 |
| `seasoningWords: Set<string>` | 菜谱食材行 `ingredientId + role`，调料也是正式食材 UUID | 菜谱详情、计划、购物清单 | 食谱策划 + 食材目录 | 删除靠字符串判断调料的规则；`role=seasoning` 明确表达“这道菜里是调料”，食材本身仍可有 `kind=condiment` |
| `Recipe.required: string[]` | `RecipeIngredientLine[]`：`ingredientId/role/quantity/unitId/preparation` | 菜谱、今日计划、购物清单 | 菜谱数据 | 先经 approved 别名解析成 UUID；解析不唯一时要求人工确认，不能保存错误 ID |
| `recognitionPresets[].foodId` 使用短 ID | `RecognitionCandidate.ingredientId` 使用固定 UUID | 图片/截图/小票录入 | 识别适配层 + legacy ID 映射 | seed 导入事务内迁移旧 ID；置信度仍是识别结果，不是主数据属性 |
| `photo` 一个 URL 或内联图 | `imageAssets[]`，并选一个 `isPrimary` | 仓库卡片、识别确认、菜谱选材 | 当前为 Miiix 自有程序化 SVG；未来可用审核后的外部素材 | 不再把无作者、无许可 URL 当已批准图片；详细门禁见图片治理文档 |
| 没有图片作者、许可和处理记录 | `originalUrl/processedUrl/providerAssetId/licenseCode/licenseUrl/rightsStatus/attribution/transformLog/hash/review*` | 素材审核；用户页面只读已发布资产 | 素材提供方 + Miiix 权利审核 | 旧 Unsplash URL 不自动升级为已批准资产；缺元数据时隔离或替换，不补造作者/许可 |
| `storage` 四个枚举值 | `defaultStorageProfileId` 指向一个储藏档案；档案引用 `storageMethodId` | 仓库分区、录入默认值 | 中国场景优先国家卫健委；海外补充按地区保存 | 从单个标签升级为可追溯档案；默认建议不能覆盖用户批次实际位置 |
| `storageTags: string[]` | `environmentTags[] + instructions` | 仓库卡片、录入提示 | [国家卫健委家庭食品安全提示](https://www.nhc.gov.cn/sps/c100087/202204/835f7abc9a7b40fe97c0a7264c9631e8.shtml) 或记录级来源 | 通用标签可复核后迁移；“低脂备餐”“去虾线”等非储藏事实移出储藏字段 |
| `shelfLifeDays: number` 单值 | `minDays/maxDays/recommendedDays + context + region + foodState + packagingState + endpoint + sourceId` | 新鲜度提示、到期建议 | 中国权威提示优先；US 补充用 [FoodKeeper CC0 数据](https://catalog.data.gov/dataset/fsis-foodkeeper-data) | 旧演示天数不继承；有记录级证据才填范围。无证据时天数为 `null`，保留有来源的文字提示 |
| `defaultMode` | `defaultAmountMode` | 手动录入、识别确认 | Miiix 交互规则 | 只表示默认输入方式，不代表真实采购规格 |
| `defaultCount`、`defaultWeight` | `defaultPurchaseQuantity: number \| null` | 手动录入的起始输入 | 暂无可靠用户研究 | 主数据保持 `null`；UI 可临时显示 1 个或 250 克作为界面起点，但不得回写成目录事实 |
| `unit` 一个中文字符串 | `defaultUnitId + supportedUnitIds[]`，单位是独立记录 | 录入、库存、购物清单 | 单位规范 + Miiix 交互 | “个/颗/根”先统一为计数维度或按展示词处理；底层使用稳定单位 ID |
| 没有结构化单位换算；页面暗含个/克 | `unitConversions[]`：from/to/factor/source/confidence/reviewStatus | 数量编辑、营养计算 | 逐食材、逐规格证据 | 克/千克等同维度用基础换算；个->克、碗->克、包->克无证据时不得创建假换算 |
| `price` 被写成食材常量 | `referencePrice = null` | 不再作为目录默认价格展示 | 本版无地区/时间/渠道价格源 | 删除演示“全国均价”含义；若 UI 需要输入，初始值为空而不是假数字 |
| `InventoryItem.pricePaid` | `InventoryLot.priceAmount + currency` | 库存详情、消费记录 | 用户真实录入或票据 | 保留为用户批次事实；目录更新不得覆盖 |
| `caloriesPer100g: number` | `nutritionProfile.caloriesKcal`，连同 `basisQuantity/basisUnitId` | 食材信息、菜谱热量估算 | 记录级营养来源 | 只有食材粒度和状态匹配且审核通过才显示；未知为 `null`，不能用 0 代替 |
| 没有蛋白质/脂肪/碳水/纤维来源信息 | `proteinG/fatG/carbohydrateG/fiberG + dataClassification + foodState + sourceRecordId + sourceRelease + matchType` | 食材信息、未来营养计算 | [USDA FoodData Central / SR Legacy](https://fdc.nal.usda.gov/download-datasets/) 或获授权的地区数据 | 所有数字绑定版本和记录号；杨梅、鸡爪、上位鱼/油等高风险错配项留空 |
| 没有目录级来源记录 | `CatalogSource`：provider、dataset、version、URL、license、jurisdiction、retrieved/reviewed | 数据审核页；业务页面用于“来源”说明 | 官方来源及 Miiix 自有数据 | 先登记来源再引用；“网上查到”不是合格来源 |
| 没有数据表达标准 | 字段与单位表达参考 `WS/T 464-2015` | 数据管理，不直接展示 | [国家卫健委 WS/T 464-2015](https://www.nhc.gov.cn/wjw/yingyang/201505/3cbe4ecd6e48465899557a25a5ae1be9.shtml) | 用于表达和治理；不能因此批量复制付费《中国食物成分表》第 6 版 |
| 没有外部记录映射 | `externalMappings[]`：provider/key/label/matchType/source/review* | 数据审核；营养证据链 | USDA、FoodKeeper、可选 TW 数据源 | 只有记录级证据与粒度审核通过才 approved；代表性/上位/下位关系不能冒充 exact |
| 没有版本和人工复核字段 | `dataVersion/createdAt/updatedAt/reviewedAt/reviewedBy/status` | 数据管理、问题追踪 | Miiix 发布流程 | seed 每次修订更新 dataVersion；归档不删除，保留历史可解释性 |
| `FoodInfo` 被完整复制进 `InventoryItem` | `InventoryLot.ingredientId` 引用主数据；批次只存数量、单位、位置、价格、日期等 | 仓库和库存详情 | 主数据 + 用户输入分工 | 迁移 ingredient/unit/storage 的技术引用；不改变用户数量与计量含义，也不批量重写位置语义、价格和到期日 |
| `addedDaysAgo` 相对天数 | `purchasedAt/openedAt/expiresAt` 绝对时间 | 新鲜度、库存详情 | 用户输入 + 可解释的默认建议 | 迁移时以基准时间换算并记录来源；后续不继续存“几天前”这种随时间漂移的值 |
| 页面按 `item.name === requiredName` 判断拥有 | 按 `ingredientId` 判断；approved 别名只用于导入解析 | 今日计划、购物清单 | CatalogRepository | 避免“番茄/西红柿”漏匹配，也避免“猪肉/猪肉末”误匹配 |
| 本地 CatalogRepository 在内存从 `foodLibrary` 拼记录 | IndexedDB v2 `catalogSources/categories/units/storageMethods/ingredients` + 只读 `CatalogRepository` | 所有需要目录的页面 | v0.4.2 有版本 seed | UI 不直接维护第二份事实；Repository 支持 ID、别名、分类、储藏、kind、status 查询 |
| kitchen demo seed 与目录隐式耦合 | `catalog-seed-version`、digest、统计与 kitchen seed 分离 | 用户无感 | 本地持久化策略 | 目录重导不会重复制造演示库存；版本/摘要相同则幂等跳过 |
| IndexedDB v1 没有目录 store | IndexedDB v2 增量新建目录 store | 离线应用启动 | schema upgrade | 保留所有 v1 业务 store；导入和 legacy 引用迁移使用一次事务，失败整体回滚 |
| PostgreSQL `0001` 的基础目录字段 | `0006` 增加科学名、默认输入、版本/审核、储藏范围、支持单位、营养证据、图片权利、外部映射来源 | 未来云端同步/数据管理 | 追加式数据库迁移 | 不修改 `0001`–`0005`；先做空库与带数据迁移演练，应用回退时保留新增列表 |

## 3. 页面读取关系

| 页面/流程 | v0.4.2 应读取 | 不应再读取或推断 |
|---|---|---|
| 仓库首页 | CatalogRepository 的食材投影 + 用户 Inventory Lot | 页面内 `foodLibrary` 第二份事实、全国均价 |
| 手动录入 | 正式名、approved 别名、三级分类、支持单位、默认储藏建议 | 用字符串猜 ID、用 UI 初始数量冒充默认采购量 |
| 图片/截图/小票确认 | 识别候选的 ingredient UUID、候选置信度、目录图片 | 模型标签直接写入正式名、未批准别名自动命中 |
| 菜谱详情 | `RecipeIngredientLine` 的 ingredientId、role、quantity、unit | `seasoningWords`、`required: string[]` 的名字匹配 |
| 今日计划 | 菜谱食材 UUID 与可用 lot 的 ingredientId | `item.name === requiredName` |
| 购物清单 | 缺少的 ingredientId、展示名和用户所需数量 | 把上位食材和具体部位视作同一项 |
| 新鲜度提示 | 用户批次日期 + 对应状态/地区的储藏档案 | 用一个全局 `shelfLifeDays` 断言食品安全 |

## 4. 空值和冲突处理

| 情况 | 正确处理 | 错误处理 |
|---|---|---|
| 找不到精确营养记录 | 数值 `null`，`reviewStatus=pending`，说明原因 | 借相似食材的值并标 approved |
| 只有 US 储藏数据 | 保存 `region=US` 的档案 | 去掉地区后展示为中国结论 |
| 用户输入“鱼” | 命中“鱼（品种未指定）”或要求确认具体鱼种 | 自动映射为鲈鱼 |
| 用户输入“酱油” | 给出生抽等候选，要求确认 | 静默保存为生抽 |
| 同一别名指向两个食材 | 阻止第二个别名 approved | 按置信度随机选一个 |
| UI 需要默认数量 | 使用明确标记为 UI fallback 的起始值 | 写进 `defaultPurchaseQuantity` 当成数据事实 |
| 没有参考价格来源 | `referencePrice=null` | 保留 v0.4.1 演示价 |
| 图片许可不完整 | 隔离、pending 或替换为自有占位图 | 补猜作者和许可后发布 |

## 5. 数据源和地区标签

营养和储藏的官方候选来源包括：

- 中国表达规范：[WS/T 464-2015](https://www.nhc.gov.cn/wjw/yingyang/201505/3cbe4ecd6e48465899557a25a5ae1be9.shtml)；
- 中国家庭储藏场景：[国家卫健委家庭食品安全提示](https://www.nhc.gov.cn/sps/c100087/202204/835f7abc9a7b40fe97c0a7264c9631e8.shtml)；
- 美国营养数据：[USDA FoodData Central API 与许可说明](https://fdc.nal.usda.gov/api-guide/)；
- 美国储藏数据：[USDA FSIS FoodKeeper（CC0）](https://catalog.data.gov/dataset/fsis-foodkeeper-data)；
- 台湾补充数据：[台湾食品药物管理署食品营养成分数据库](https://www.fda.gov.tw/TC/site.aspx?r=2022294925&sid=271)，使用时明确标 `TW` 并复核具体授权。

《中国食物成分表》第 6 版是付费出版物，不纳入可批量复制的 seed 来源。购买书籍不等于取得数据库再发布权。

## 6. 迁移验收清单

以下是应验证的结果，不是本文对执行状态的声明：

- 旧 18 项短 ID 均有唯一去向，且 v0.4.1 已知业务结构中的引用同步更新；任意未来 JSON 元数据字段不被假定为自动可迁移；
- 30 项 UUID、正式名和 legacy 映射在重复 seed 后保持不变；
- 所有 Inventory Lot 的数量、计量含义、价格、日期、位置语义和备注在升级前后相同；单位技术 ID 可以迁移到稳定 UUID，但不能改变原数量代表的单位；
- approved 别名索引无冲突，含混上位词用例不发生静默窄化；
- 页面通过 CatalogRepository 或其受控投影读取食材，不再从第二份硬编码数组获取事实；
- 未获证据的数据继续为 `null/pending`，不会因迁移被自动补成 0 或演示值。
