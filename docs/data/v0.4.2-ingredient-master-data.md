# Miiix v0.4.2 食材主数据标准

> 版本：v0.4.2
> 唯一范围：把食材目录从页面里的演示常量，收敛成可追溯、可校验、可迁移的主数据。
> 本文定义目标和验收口径；2026-07-19 的实际执行证据见 `outputs/v0.4.2-ingredient-catalog-self-check.md`。仓库自动化、生产构建和 390 × 844 浏览器验收已通过，PostgreSQL 实库演练仍是云端部署前门禁。

## 1. 这一版只做什么

v0.4.2 只完成“食材主数据”这一件事：统一食材身份、三级分类、别名、储藏建议、单位、营养来源、图片资产、外部映射和审核状态，并让本地 IndexedDB 与未来 PostgreSQL 使用同一套语义。

这一版明确不做：

- 不接入新的 AI 识别、生成菜谱、RAG、向量检索或模型训练；
- 不把 30 项称为“全国最常见 Top 30”，因为本版没有全国消费频次数据支持这种结论；
- 不建设品牌、条码、商超 SKU、产地批次或实时价格库；
- 不伪造食材保质期、营养、个数与克数换算、全国均价或图片许可；
- 不扩成 200 项后再回头补规则，规则必须先在 30 项上跑通。

“30 项”是工程黄金样本，不是市场排名。它们被选择来覆盖旧数据兼容、常见家庭库存、三级分类、同义词、含混上位词、生熟状态、计数/重量、主料/调料和空值处理等难点。

## 3. 为什么是这 30 项

选择依据按优先级排列：

1. 先覆盖 v0.4.1 已经出现在库存、识别预设、菜谱和购物清单里的食材，避免升级后旧记录失联；
2. 再覆盖用户主路径：录入库存、按食材检索、推荐菜谱、生成购物清单、记录购买批次；
3. 有意加入难例：猪肉/猪肉末、鱼/具体鱼种、大米（生）/米饭（熟）、食用油/具体油种；
4. 覆盖 `raw`、`processed`、`beverage`、`dish_component`、`condiment` 五种食材类型；
5. 覆盖计数、重量、体积、包装单位，但不借此制造跨维度换算。

本版排除品牌牛奶、不同脂肪率牛奶、具体猪牛部位、具体鱼虾品种、蜂蜜、醋、菌菇、香辛料、水和冰等。它们并非不重要，而是当前没有必要在同一版本扩宽边界。水和冰也可能是“厨房公用资源”而不是库存食材，必须先做产品语义决策。

## 4. 扩展到 200 项之前的门槛

以下门槛必须同时满足，不能用“先录完再清洗”替代：

- 30 项的固定 UUID、正式名、三级分类、默认单位、默认储藏档案、来源和审核字段完整率为 100%；
- 主数据结构校验、引用完整性校验、固定 ID 校验和跨表唯一性校验能够自动执行；
- 同一 `locale + normalizedAlias` 的已批准别名冲突为 0；含混上位词误自动匹配为 0；
- 每个显示出来的营养数字都有食物状态、版本、来源记录号和匹配类型；没有精确证据的项可为空，但不得借值；
- 每个储藏建议都标明地区、食物状态、包装/开封状态、范围终点和来源；没有可靠天数时允许只有文字建议；
- 跨维度单位换算为 0 个无证据条目；`个 -> 克`、`包 -> 克` 必须逐食材、逐规格审核；
- 30 项图片均通过素材发布门禁；原型占位图不得被误标为实拍图；
- IndexedDB v1 到 v2 的升级、重复导入、失败回滚和旧 ID 引用迁移具有可重复的验证证据；
- PostgreSQL `0006` 在空库和带 v0.4.1 数据的库上完成迁移演练，并保留用户 Inventory Lot；
- 真实录入样本显示未命中别名率可接受，且新增别名有人工审核流程；
- 有明确的数据负责人、复核人和版本发布节奏，否则 200 项只会放大错误。

## 5. 主数据结构

### 5.1 目录级字段

| 字段 | 规则 | 给非技术用户的解释 |
|---|---|---|
| `schemaVersion` | 结构变化时更新 | “表格列和规则”的版本 |
| `catalogVersion` | 与产品数据发布对应，如 `0.4.2` | 这批目录属于哪个产品版本 |
| `dataVersion` | 每次数据修订唯一且单调变化 | 同一结构下第几次数据修订 |
| `publishedAt` / `reviewedAt` | ISO 8601，含时区 | 何时发布、何时复核 |
| `reviewedBy` | 不可为空 | 谁为这一版数据负责 |
| `sources` | 先登记来源，再允许记录引用 | 任何数字或规则都能追溯 |

### 5.2 食材身份

每个食材必须有 `id`、`slug`、`canonicalNameZh`、`kind`、三级分类 ID、`defaultUnitId`、`supportedUnitIds`、`defaultStorageProfileId`、`status`、`dataVersion`、审核人和审核时间。

- UUID 是系统身份，名称只是可修改的展示属性；UUID 永不复用。
- `legacyIds` 是旧记录迁移映射。一个旧 ID 只能指向一个新 UUID，且已发布映射不可换目标。
- 例如旧 `pork` 在 v0.4.1 实际代表“肉沫”，因此兼容映射到“猪肉末”，不能因为英文词义较宽就改映射到“猪肉（部位未指定）”。
- `scientificName` 只在物种粒度确定时填写。鱼、虾、食用油、豆腐等上位项不得伪填某个具体物种或产品。
- `status` 使用 `draft / active / archived`。归档不等于删除，历史 Inventory Lot 仍要能解释原食材。

### 5.3 三级分类

所有 active 食材必须且只能落到一条完整路径：一级 -> 二级 -> 三级。v0.4.2 的一级域为：

- 肉禽蛋奶豆及水产；
- 蔬果；
- 主食；
- 粮油调味。

分类 ID 与名称分离。移动分类只改父子关系，不能改食材 UUID。一级、二级、三级的 `categoryIds` 顺序固定；校验器必须检查父子链真实相连，而不只是三个 ID 都存在。

### 5.4 别名与冲突

别名类型为：

- `common`：通用同义词，如“马铃薯”；
- `regional`：地区称呼，如“青瓜”；
- `retail`：零售包装或售卖称呼；
- `ocr`：票据/OCR 常见变体；
- `translation`：跨语言名称；
- `external_model`：外部识别模型标签；
- `recipe_phrase`：食谱短语，如“少量糖”。

别名先做 Unicode NFKC、去首尾空白、统一大小写和标点，得到 `normalizedAlias`。只有 `reviewStatus=approved` 的别名可以自动解析。

冲突规则：

1. 同一语言下，一个已批准标准化别名只能指向一个食材；
2. “鱼”“猪肉”“辣椒”“酱油”等上位词不能静默解析到更窄的品种或部位；
3. 出现一对多候选时返回“需要用户确认”，不能按置信度偷偷选一个；
4. 区域别名必须保留 `locale/region`，跨地区不得默认复用；
5. 搜索可以展示相关候选，写入 Inventory Lot 前必须落到唯一 UUID。

### 5.5 储藏档案

“冷藏 7 天”不是一个脱离上下文的事实。每个 `storageProfile` 必须同时记录：

- `storageMethodId`：阴凉通风、冷藏、冷冻或干燥避光等物理环境；“调味品”是食材类型或菜谱角色，不是储藏方式；
- `region`：至少区分 `CN`、`US`、`TW`；
- `foodState`：生、熟、带壳、切开、品种未指定等；
- `packagingState`：未开封、已开封、散装、密封分装或遵循包装标签；
- `environmentTags`：干燥、避光、远离热源、生熟分开等；
- `minDays / maxDays / recommendedDays`：有证据才填，且最小值不得大于最大值；
- `endpoint`：天数表示安全、品质，还是“遵循包装标签”；
- `instructions`、`sourceId`、`evidenceKey`、置信度和审核记录。

每个食材只有一个 `defaultStorageProfileId`，但可以有多个不同状态的档案。默认值是录入建议，不是用户购买批次的强制到期日。海外来源只能标相应地区，不能伪装成中国家庭场景结论。

### 5.6 单位和默认购买量

- `supportedUnitIds` 表示允许用户怎样记录，不表示单位之间可以互换。
- 同维度基础换算（克/千克、毫升/升）可由单位表提供。
- 跨维度换算（个/克、碗/克、包/克、毫升/克）必须逐食材提供 `unitConversions`、来源、置信度和审核状态；v0.4.2 没有可靠依据时数组为空。
- `defaultPurchaseQuantity` 允许为 `null`。UI 为方便输入给出的 1 个或 250 克只是界面起始值，不是主数据事实。
- 包装规格属于具体商品或用户购买批次，不能写成所有食材的固定属性。

### 5.7 营养档案

营养数字必须绑定：计量基准、食物状态、数据分类、来源版本、来源记录号、匹配类型、审核状态和审核人。

- 基准统一明确为“每 100 克”时，仍需保存 `basisQuantity=100` 和克单位 ID；
- `dataClassification` 区分分析值、计算值、估算值和未知；
- `matchType` 区分精确、代表性、更宽、更窄或无匹配；
- 生/熟、带皮/去皮、部位、品种和产品类型必须与来源记录一致；
- 杨梅、鸡爪、鸡腿肉、上位“鱼”、上位“食用油”等高风险错配项，在没有粒度相符的证据时保持 `null + pending`；
- 0 是测量值，`null` 是未知，两者不能互换。

### 5.8 价格

30 项的 `referencePrice` 一律为 `null`。全国均价会随城市、日期、渠道、规格和品质剧烈变化，本版没有可靠的时间序列价格来源，不能把演示数字包装成事实。

用户实际支付的金额保存在 Inventory Lot 的 `priceAmount`，它是用户事实，不是食材主数据。未来如做参考价，应建立独立的“地区 + 渠道 + 日期 + 规格 + 来源”的价格观察表，不得回填成食材的单个常量。

### 5.9 外部映射

外部映射必须包含外部系统、记录号、外部名称、来源、匹配类型、审核状态、审核人和时间。只有同时满足以下条件，才可标记 `approved` 并被营养或储藏数据引用：

1. 找到可复查的记录级证据；
2. 食材种类、部位/品种、加工状态和生熟状态与本地粒度一致；
3. 来源版本固定；
4. 人工复核通过。

“鱼（品种未指定） -> 鲈鱼”“食用油 -> 大豆油”“豆腐（类型未指定） -> 北豆腐”都不是精确映射。可作为候选或代表性关系保存，但不得借此显示精确营养数字。

## 6. 来源优先级与许可边界

来源先按场景选，再按权威性选；不是把所有来源混成一个“真值”。

1. 表达和字段规范优先参考国家卫健委 [WS/T 464-2015《食物成分数据表达规范》](https://www.nhc.gov.cn/wjw/yingyang/201505/3cbe4ecd6e48465899557a25a5ae1be9.shtml)。该标准用于结构和表达要求，不自动授权复制其他付费数据库，也不直接提供本项目所需的全部数值。
2. 中国家庭储藏和安全语境优先采用国家卫健委 [家庭食品安全提示](https://www.nhc.gov.cn/sps/c100087/202204/835f7abc9a7b40fe97c0a7264c9631e8.shtml)。提示明确区分常温、冷藏 0–5℃、冷冻 -18℃，并强调生熟分开；仅把文中明确覆盖的食材和状态写成具体规则。
3. 营养值可采用 USDA [FoodData Central](https://fdc.nal.usda.gov/api-guide/) 或其 [SR Legacy / 下载数据](https://fdc.nal.usda.gov/download-datasets/)。USDA 说明 FDC 数据为美国公有领域并以 CC0 发布；仍必须保存发布版本、记录号和食物状态，且不得把美国特定品种冒充中国上位食材。
4. 海外储藏补充可采用 USDA FSIS [FoodKeeper 数据集](https://catalog.data.gov/dataset/fsis-foodkeeper-data)。该数据集标为 CC0；使用时标 `region=US`，中国来源与它冲突时不直接覆盖，而是保留两个地区档案。
5. 台湾食品药物管理署 [台湾地区食品营养成分数据库](https://www.fda.gov.tw/TC/site.aspx?r=2022294925&sid=271) 可作补充，但所有记录必须标 `TW`，并在批量使用前复核下载项的具体授权条款。
6. 《中国食物成分表》第 6 版属于付费出版物。本项目不能未经许可批量复制整表、截图或把大量数值提交到仓库；若团队合法购买，也只用于人工核对，批量入库需要单独取得授权并保存授权凭据。

来源的 `license` 不能为空白地写“可用”。不确定时写明“来源条款适用，未主张开放许可”，并限制为结构参考或人工核对。

## 7. 本地 IndexedDB v2 策略

IndexedDB v2 负责浏览器离线读模型：

- v1 原有库存、菜谱、计划、日志等业务 store 保持不删；
- v2 新增来源、分类、单位、储藏方式和食材聚合 store；
- 黄金样本作为有版本的 seed，经一次事务写入；meta 保存 `catalogVersion`、内容摘要和导入统计；
- seed 版本与厨房演示数据版本分开，更新目录不能重复插入演示库存；
- 同版本同摘要重复启动时不重写；版本或摘要变化时以一次事务替换目录快照；
- 同一事务迁移 v0.4.1 已知结构中的旧 ingredient、unit 和 storage method 引用，任一步失败则目录、引用迁移和 meta 一起回滚；未来新增任意 JSON 引用字段时必须追加显式迁移；
- `CatalogRepository` 是业务层唯一读取入口，支持按 ID、批准别名、分类、储藏方式、类型和状态查询；UI 投影只负责展示兼容，不再拥有第二份食材事实。

浏览器数据库版本不能安全“降级”。如果前端版本回滚，旧应用应容忍 v2 的额外 store；需要恢复时保留 v2 数据库并关闭新入口，而不是删除整个 IndexedDB。任何重建动作前都要先导出用户业务数据。

## 8. PostgreSQL 0006 策略

`0006_catalog_data_quality.sql` 必须是追加迁移，不修改 `0001`–`0005` 的历史文件。迁移目标是：

- 给食材、别名、储藏、营养、图片、外部映射补足版本、来源、审核和上下文字段；
- 为支持单位建立关联表，不把数组塞进单个文本字段；
- 用唯一索引阻止 active 正式名冲突和 approved 别名冲突；
- 用范围约束阻止负天数和 `min > max`；
- 只向满足审核与版权门禁的资产开放目录读取策略；
- 不把 seed 中的公共默认值写进用户 Inventory Lot，也不覆盖用户支付价和到期日。

SQL 文件使用事务，部署前在空库和 v0.4.1 数据副本上演练。若迁移失败，数据库事务整体回滚；若已经成功上线后需要应用回退，优先保留新增列和表，由旧应用忽略它们，避免用破坏性 down migration 删除数据。

## 9. Inventory Lot 边界

主数据回答“这是什么食材、通常如何描述”；Inventory Lot 回答“这个用户买了哪一批、还有多少、放在哪里”。边界如下：

| 主数据 | Inventory Lot 用户事实 |
|---|---|
| 正式名、别名、分类、默认单位 | 本批次实际单位与数量 |
| 储藏建议和来源 | 本批次实际储藏位置 |
| 默认储藏档案 | 用户填写/确认的到期日、开封日 |
| 营养参考及版本 | 用户不会被目录更新强制改写的历史记录 |
| `referencePrice=null` | 本批次真实 `priceAmount` |

目录升级允许把旧 ingredient、unit 和 storage method 的技术 ID 迁移到稳定 UUID，并把旧 `seasoning` 位置代码等义改名为 `dryDark`；不得改变数量/计量含义、价格、购买时间、实际储藏语义或到期日。用户手工纠正始终优先于默认建议。
